# Story 1.3: Implement Mock API Layer, Shared Utilities, and Seed Data

Status: ready-for-dev

## Story

As a developer,
I want the mock API layer and seed data in place,
So that all task mutations can be simulated with 2-second latency and 10% failure rate during development and testing.

## Acceptance Criteria

1. **Given** `src/api/mock-client.ts` **When** `mockRequest<T>` is implemented **Then** it waits approximately 2 seconds (via `sleep(2000)`) before resolving **And** it throws `MockApiError` approximately 10% of the time (`Math.random() < 0.1`) **And** `MockApiError` extends `Error` with `name = 'MockApiError'`

2. **Given** `src/api/tasks.ts` **When** `createTask`, `updateTask`, `deleteTask`, and `moveTask` are implemented **Then** each calls through `mockRequest<T>` and returns the appropriate typed response **And** no component or hook imports `mockRequest` directly — only through `src/api/tasks.ts`

3. **Given** `src/shared/utils/sleep.ts` **When** imported **Then** it exports a `sleep(ms: number): Promise<void>` utility

4. **Given** `src/shared/utils/mockData.ts` **When** the module is loaded **Then** it exports an array of exactly 25 `Task` objects with realistic titles, descriptions, assignees, priorities, statuses (spread across 'todo', 'in-progress', 'done'), and ISO 8601 `createdAt` strings **And** each task has a unique `string` ID generated by `nanoid()`

5. **Given** `src/shared/components/ErrorBoundary.tsx` **When** a child component throws a render error **Then** the ErrorBoundary catches it and renders a fallback UI without crashing the full application

6. **Given** `src/shared/components/ToastProvider.tsx` **When** rendered **Then** it mounts the Sonner `<Toaster>` with position `"bottom-right"` and `richColors`

### Task 2 — Exact `mock-client.ts` Implementation

File: `src/api/mock-client.ts`

```typescript
// src/api/mock-client.ts
import { sleep } from '@/shared/utils/sleep'

export class MockApiError extends Error {
  constructor(message = 'Simulated API failure') {
    super(message)
    this.name = 'MockApiError'
  }
}

export async function mockRequest<T>(
  fn: () => T,
  opts = { delay: 2000, failureRate: 0.1 }
): Promise<T> {
  await sleep(opts.delay)
  if (Math.random() < opts.failureRate) throw new MockApiError()
  return fn()
}
```

Key constraints:
- `MockApiError.name` must be set explicitly in the constructor because TypeScript class transpilation does not preserve the class name reliably in all targets.
- `mockRequest` accepts a factory function `fn: () => T` rather than a resolved value to avoid computing the result before the simulated delay completes.
- `opts` has a default value at the parameter level — individual properties cannot be overridden selectively without providing the full object. This is intentional for this story's scope.
- Do NOT export `MockApiError` from `tasks.ts` — it must remain an implementation detail of `mock-client.ts`. Callers catching errors should check `error instanceof MockApiError` or `error.name === 'MockApiError'`.

### Task 3 — `tasks.ts` Implementation Guidance

File: `src/api/tasks.ts`

This file is the **sole public interface** for all task mutations. No component, hook, or context may import `mockRequest` or `MockApiError` directly.

```typescript
// src/api/tasks.ts
import { nanoid } from 'nanoid'
import type { Task, TaskStatus } from '@/types/task.types'
import { mockRequest } from './mock-client'

export async function createTask(
  task: Omit<Task, 'id' | 'createdAt'>
): Promise<Task> {
  return mockRequest<Task>(() => ({
    ...task,
    id: nanoid(),
    createdAt: new Date().toISOString(),
  }))
}

export async function updateTask(
  taskId: string,
  changes: Partial<Omit<Task, 'id' | 'createdAt'>>
): Promise<Task> {
  // The caller (BoardAPIContext) holds the current task snapshot.
  // tasks.ts receives the full merged result to return after the mock delay.
  // Pattern: caller builds the optimistic merged task, passes it here.
  return mockRequest<Task>(() => ({
    id: taskId,
    createdAt: new Date().toISOString(), // placeholder — real impl uses snapshot
    ...changes,
  } as Task))
}

export async function deleteTask(taskId: string): Promise<void> {
  return mockRequest<void>(() => undefined)
}

export async function moveTask(
  taskId: string,
  newStatus: TaskStatus
): Promise<Task> {
  return mockRequest<Task>(() => ({
    id: taskId,
    status: newStatus,
  } as Task))
}
```

**Implementation note on `updateTask` and `moveTask`:** The actual merged task must be passed in by the caller (BoardAPIContext) which holds the current state snapshot. The pattern shown above is a minimal placeholder. The canonical implementation for `updateTask` should accept the **full merged task** to return, not just the partial changes:

```typescript
// Preferred signature for updateTask — caller provides the pre-merged task:
export async function updateTask(mergedTask: Task): Promise<Task> {
  return mockRequest<Task>(() => mergedTask)
}
```

Discuss with the team which signature is preferred before implementing. Either way, `mockRequest` must be the only place the simulated delay and failure logic lives.

**Forbidden pattern (AC #2 enforcement):**
```typescript
// WRONG — never do this in any component or hook:
import { mockRequest } from '@/api/mock-client'

// CORRECT — always go through tasks.ts:
import { createTask } from '@/api/tasks'
```

### Task 5 — `ErrorBoundary.tsx` Full Implementation

File: `src/shared/components/ErrorBoundary.tsx`

React error boundaries **must** be class components. There is no functional component equivalent in React 18. The `getDerivedStateFromError` and `componentDidCatch` lifecycle methods are only available on class components.

```typescript
// src/shared/components/ErrorBoundary.tsx
import React from 'react'

type ErrorBoundaryProps = {
  children: React.ReactNode
  fallback?: React.ReactNode
}

type ErrorBoundaryState = {
  hasError: boolean
  error: Error | null
}

export class ErrorBoundary extends React.Component<
  ErrorBoundaryProps,
  ErrorBoundaryState
> {
  constructor(props: ErrorBoundaryProps) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  static getDerivedStateFromError(error: Error): ErrorBoundaryState {
    return { hasError: true, error }
  }

  componentDidCatch(error: Error, info: React.ErrorInfo): void {
    console.error('[ErrorBoundary] Caught render error:', error, info)
  }

  render(): React.ReactNode {
    if (this.state.hasError) {
      if (this.props.fallback != null) {
        return this.props.fallback
      }
      return (
        <div
          role="alert"
          style={{
            display: 'flex',
            flexDirection: 'column',
            alignItems: 'center',
            justifyContent: 'center',
            padding: '2rem',
            textAlign: 'center',
          }}
        >
          <h2>Something went wrong</h2>
          {import.meta.env.DEV && this.state.error != null && (
            <pre style={{ color: 'red', marginTop: '1rem', fontSize: '0.875rem' }}>
              {this.state.error.message}
            </pre>
          )}
        </div>
      )
    }
    return this.props.children
  }
}
```

**Critical notes:**
- `import.meta.env.DEV` is the Vite idiom for checking development mode (do not use `process.env.NODE_ENV`).
- `this.state.error != null` is the correct null-check under `noUncheckedIndexedAccess` and strict mode (avoids implicit `any` warnings on optional chaining).
- The fallback UI uses inline `style` here for simplicity — if Tailwind classes are preferred, apply them after Tailwind is confirmed available. Using inline styles makes the fallback resilient to CSS failures.
- `role="alert"` on the fallback container satisfies FR21 (graceful recovery) and aligns with accessibility requirements.
- Named export only — no `export default`.

### Optimistic Update Sequence (Context for Stories 2+)

This story implements step 4 of the optimistic update pattern. The full sequence, established here and consumed in Story 2+:

```
1. opId = nanoid()
2. Record snapshot in pendingOps: { taskId, snapshot: currentTask }
3. Apply change to tasks[] immediately (optimistic, before API call)
4. Call the appropriate fn from src/api/tasks.ts  ← THIS STORY
5a. On success → dispatch OP_SUCCESS(opId)
5b. On failure → dispatch OP_ROLLBACK(opId) + toast.error(...)
```

`tasks.ts` is the network boundary. `BoardAPIContext` (Story 1.2) is the dispatch boundary. No component ever crosses either boundary directly.
